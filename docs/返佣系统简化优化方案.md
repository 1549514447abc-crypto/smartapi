# 返佣系统简化优化方案

## 一、当前问题分析

### 1.1 存储位置混乱（5个地方存佣金配置！）

| 存储位置 | 字段 | 用途 | 问题 |
|---------|------|------|------|
| `system_configs` 表 | `commission_rate`, `commission_rate_course`, `commission_rate_membership` | 全局默认佣金 | 与 user_categories 功能重复 |
| `commission_settings` 表 | `commission_rate` 等 | 另一套全局配置 | 完全多余，未被使用 |
| `user_categories` 表 | `default_course_rate`, `default_membership_rate` | 各分类的佣金比例 | ✅ 这是正确的设计 |
| `users` 表 | `custom_course_rate`, `custom_membership_rate` | 用户自定义佣金 | 过度设计，增加复杂性 |
| 代码硬编码 | `DEFAULT_CONFIG` | 最终兜底值 | 不应该存在 |

### 1.2 代码重复定义

```
CommissionService.ts:
  - getCommissionRate()
  - getCourseCommissionRate()
  - getMembershipCommissionRate()
  - getCommissionRateForUser()

referralController.ts:
  - getCourseCommissionRate()      // 重复！
  - getMembershipCommissionRate()  // 重复！
  - getGeneralCommissionRate()     // 重复！
```

### 1.3 Fallback 逻辑过于复杂

当前 `getCommissionRateForUser()` 的查找顺序：
```
1. users.custom_course_rate (用户自定义)
   ↓ 如果为空
2. user_categories.default_course_rate (用户分类)
   ↓ 如果为空
3. system_configs.commission_rate_course (系统配置)
   ↓ 如果为空
4. DEFAULT_CONFIG (代码硬编码 10%)
```

**问题：** 4层fallback，任何一层出问题都难以排查，后台设置的值可能被覆盖。

### 1.4 涉及的页面清单

#### 前端用户页面（3个）

| 页面路径 | 文件 | 涉及内容 | 是否需要改 |
|---------|------|----------|-----------|
| `/referral` | `frontend/src/pages/Referral/index.tsx` | 显示佣金比例（从API获取） | ❌ 不需要 |
| `/commission` | `frontend/src/pages/Commission/index.tsx` | 显示佣金比例、提现功能 | ❌ 不需要 |
| `/membership` | `frontend/src/pages/Membership/index.tsx` | 会员购买（不涉及返佣） | ❌ 不需要 |

#### Admin 后台页面（4个涉及）

| 页面 | 文件 | 涉及内容 | 是否需要改 |
|------|------|----------|-----------|
| 分佣设置 | `admin/src/pages/UserCategories/index.tsx` | 管理用户分类和各分类佣金比例 | ✅ **保留，这是正确的** |
| 价格管理 | `admin/src/pages/Pricing/index.tsx` | 设置普通会员佣金比例 | ⚠️ 需简化：删除system_configs相关 |
| 用户管理 | `admin/src/pages/Users/index.tsx` | 编辑用户佣金（自定义比例） | ⚠️ 需简化：删除自定义比例功能 |
| 提现管理 | `admin/src/pages/Withdrawals/index.tsx` | 提现审核（不涉及比例设置） | ❌ 不需要 |

## 二、简化后的设计

### 2.1 核心原则

**一个用户只看一个分类，一个分类对应固定的佣金比例**

```
用户 → 用户分类 → 该分类的佣金比例
```

### 2.2 数据表设计（保留2张表）

#### `user_categories` 表（保留，唯一的佣金配置来源）
| 字段 | 说明 |
|------|------|
| `category_key` | 分类标识 (normal, blogger, vip) |
| `category_name` | 分类名称 |
| `default_course_rate` | 课程佣金比例 (%) |
| `default_membership_rate` | 会员佣金比例 (%) |

#### `users` 表（保留 user_category 字段）
| 字段 | 说明 |
|------|------|
| `user_category` | 用户所属分类，默认 'normal' |

### 2.3 简化后的查询逻辑

```typescript
async getCommissionRateForUser(userId: number, orderType: 'course' | 'membership'): Promise<number> {
  // 1. 获取用户
  const user = await User.findByPk(userId);
  if (!user) throw new Error('用户不存在');

  // 2. 获取用户分类（默认 normal）
  const categoryKey = user.user_category || 'normal';

  // 3. 查询分类的佣金比例
  const category = await UserCategory.findOne({
    where: { category_key: categoryKey, is_active: true }
  });

  if (!category) throw new Error(`用户分类 ${categoryKey} 不存在`);

  // 4. 返回对应比例
  return orderType === 'course'
    ? Number(category.default_course_rate) / 100
    : Number(category.default_membership_rate) / 100;
}
```

## 五、简化后的管理后台操作

### 5.1 设置佣金比例（两种方式都可以）

**方式一：通过「分佣设置」页面**（推荐）
- 路径：后台 → 分佣设置 → 编辑对应分类
- 直接修改 `normal`（普通用户）、`blogger`（博主）、`vip`（关系户）等分类的佣金比例
- 这是唯一的佣金配置来源

**方式二：通过「价格管理」页面**
- 只能修改「普通会员」的默认佣金比例
- 实际保存到 `user_categories` 表的 `normal` 分类

### 5.2 修改用户分类

**通过「用户管理」页面**
- 点击用户的「佣金」按钮
- 只需选择用户分类（normal/blogger/vip 等）
- 不再支持自定义比例，用户的佣金由分类决定

### 5.3 新增特殊佣金分类

如果需要给某个用户特殊的佣金比例：
1. 先到「分佣设置」新增一个分类（如 `special_50`，佣金50%）
2. 再到「用户管理」把该用户的分类改为 `special_50`

## 六、数据库迁移脚本

```sql
-- 1. 确保 user_categories 表存在并有默认分类
-- （运行 init-user-categories.js 脚本）

-- 2. 删除多余的表
DROP TABLE IF EXISTS commission_settings;

-- 3. 删除 users 表的多余字段（可选，保留也不影响）
ALTER TABLE users DROP COLUMN IF EXISTS custom_course_rate;
ALTER TABLE users DROP COLUMN IF EXISTS custom_membership_rate;
ALTER TABLE users DROP COLUMN IF EXISTS commission_note;

-- 4. 删除 system_configs 中的佣金配置
DELETE FROM system_configs WHERE config_key IN (
  'commission_rate',
  'commission_rate_course',
  'commission_rate_membership'
);

-- 5. 确保所有用户都有分类
UPDATE users SET user_category = 'normal' WHERE user_category IS NULL;
```

## 七、简化后的数据流

### 7.1 后台设置佣金
```
管理员在"价格管理"页面设置佣金比例
    ↓
保存到 user_categories 表对应分类
    ↓
完成
```

### 7.2 前端显示佣金比例
```
用户访问推广页面
    ↓
调用 /commission/stats API
    ↓
CommissionService.getCommissionRateForUser()
    ↓
查询 users.user_category → user_categories 表
    ↓
返回佣金比例
```

### 7.3 计算实际佣金
```
用户A推荐的用户B购买课程
    ↓
CommissionService.createCommission()
    ↓
获取用户A的分类 → 获取该分类的课程佣金比例
    ↓
计算佣金金额 = 订单金额 × 佣金比例
    ↓
创建佣金记录
```
